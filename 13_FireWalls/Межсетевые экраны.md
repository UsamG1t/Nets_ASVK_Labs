## Межсетевые экраны

_Межсетевой экран_ — FireWall — особая программная (или аппаратно-программная) технология обработки сетевого трафика. Межсетевые экраны работают на всех уровнях стека протоколов,  и с помощью описываемых в них правил занимаются **ограничением** (контроль количества и скорости поступления трафика), **перенаправлением** (`DNAT`, `Port Forwarding`), **преобразованием** (в самом частом случае трансляцией (`SNAT`) ) и **учётом** трафика на любом этапе его взаимодействия с устройством.

---

### Межсетевой экран NFTables

Для разных уровней стека протоколов существуют разные межсетевые экраны. В данной работе рассматривается универсальный межсетевой экран [`nftables`](https://wiki.nftables.org/). Его особенность заключается в покрытии наибольшего числа уровней стека протоколов, а также расширенные возможности обработки трафика за счёт работы специальной виртуальной машины, анализирующей задаваемые правила и выполняющей их.

Схема работы межсетевого экрана представлена на картинке:

![](nf-hooks.png)

В рамках лабораторной рассматривается `IP Layer` - уровень обработки, соответствующий сетевому и транспортному уровням стека протоколов.

### Особенности структуры NFTables

В зависимости от типа поступающего трафика он может обрабатываться на разных [контрольных точках](https://wiki.nftables.org/wiki-nftables/index.php/Netfilter_hooks) ("хуках") сервиса.

Все поступающие пакеты первоначально проходят контрольную точку `Prerouting`, после чего проверяется адресат пакета. В случае обработки пакета своего сервиса производится передача его приложению через контрольную точку `Input`. При маршрутизации пакета он проходит контрольную точку `Forward`, после чего отправляется, на выходе проходя контрольную точку `Postrouting`. При отправке пакета из приложения он проходит контрольную точку `Output` и также отправляется, проходя `Postrouting`.

Все правила представляют из себя некоторые единичные действия над пакетами, точками входа-выхода трафика, некоторыми статически или динамически собранными множествами и т.д.

![](nf-chains.png)

Однотипные правила, решающие общую подзадачу, объединяются в **цепочки** и выполняются последовательно друг за другом. При выполнении некоторых условий возможно прерывать исполнение цепочки для, например, перехода к следующей цепочке или выполнения другой цепочки с последующим возвратом в текущую.

Цепочки привязываются к контрольным точкам и выполняются последовательно (согласно приоритетам) в рамках обработки одной контрольной точки. И также, как и для цепочек, существуют правила перехода к следующим контрольным точкам. Цепочки можно объединять в **таблицы** с целью работы с ними, как с единым объектом .

### Базовая настройка сети

Для настройки сетей воспользуйтесь [Systemd](https://ru.wikipedia.org/wiki/Systemd), аналогично настройке прошлой лабораторной. С помощью конфигурационных файлов задайте:
 + Адреса на соответствующих интерфейсах;
 + Пробрасывание пакетов между интерфейсами (IPv4Forwarding);

Также для абонента srv необходимо включить автоконфигурирование сети на интерфейсе eth0, через который осуществляется соединение с внешней сетью. За конфигурирование сети отвечает протокол DHCP. Он автоматически настраивает DNS, трансляцию адресов для srv, позволяя обращаться с него в глобальную сеть.

:round_pushpin: 1. Скопируйте или перепишите конфигурационные файлы настройки в соответствующие виртуальные машины.

`@srv`:`/etc/systemd/network/10-inet.network`
```systemd
[Match]
Name=eth0

[Network]
DHCP=ipv4
IPv4Forwarding=yes
```

`@srv`:`/etc/systemd/network/50-intnet.network`
```systemd
[Match]
Name=eth1

[Network]
Address=10.0.1.1/24
IPv4Forwarding=yes
```

`@client:`/etc/systemd/network/50-intnet.network`
```systemd
[Match]
Name=eth1

[Network]
Address=10.0.1.254/24

[Route]
Gateway=10.0.1.1
Destination=0.0.0.0/0
```

:round_pushpin: 2. На каждой виртуальной машине включите службу `systemd-networkd`

`srv`
```console
[root@srv ~]# systemctl enable --now systemd-networkd
[root@srv ~]#
```

`client`
```console
[root@client ~]# systemctl enable --now systemd-networkd
[root@client ~]#
```

## Настройка NFTables

:round_pushpin: 1. На srv включите службу `nftables`

`srv`
```console
[root@srv ~]# systemctl enable --now nftables.service
Created symlink '/etc/systemd/system/multi-user.target.wants/nftables.service' -> '/usr/lib/systemd/sys
tem/nftables.service'.
[root@srv ~]#
```

:round_pushpin: 2. С помощью команды `nft list ruleset` выведите текущий вид таблиц межсетевого экрана

`srv`
```console
[root@srv ~]# nft list ruleset
table inet filter {
       chain input {
               type filter hook input priority filter; policy accept;
       }

       chain forward {
               type filter hook forward priority filter; policy accept;
       }

       chain output {
               type filter hook output priority filter; policy accept;
       }
}
[root@srv ~]#
```

:information_source: По умолчанию в системе присутствует единственная таблица с именем `filter` (имя таблицы может быть любым и никак не связано с типом входящих в него цепочек), цепочки из которой применяются для пакетов всех типов (ключевое слово `inet` используется для цепочек, подходящих под все пакеты, слова `ip` и `ipv6` - только для пакетов соответствующих протоколов).

:information_source: Таблица содержит три пустые цепочки (имена цепочек могут быть любыми и никак не связаны с контрольными точками, куда они привязываются). Для каждой цепочки задаются тип правил в ней (`nat` - преобразование, `filter` - фильтрация, `route` - перенаправление), соответствующий хук (если он есть), а также приоритет цепочки с помощью ключевых слов. Также следует явно указывать политику использования цепочки (некоторые могут иметь явный `reject` и не обрабатывать трафик до наступления некоторых условий или явной смены политики).

### Настройка трансляции адресов

В прошлой лабораторной мы упоминали, что при настройке выхода в интернет с помощью DHCP для настраиваемого узла автоматически включается трансляция адресов. В данном разделе необходимо включить трансляцию адресов для внутреннего абонента сети client.

:round_pushpin: 1. С помощью _команд мониторинга сети_ проверьте невозможность выхода в интернет с абонента client:

`@client`
```console
[root@client ~]# ping -c5 77.88.44.242
PING 77.88.44.242 56(84) bytes of data.
^C
--- 77.88.44.242 ping statistics ---
4 packets transmitted, 0 received, 100% packet loss, time 4005ms
[root@client ~]#
```

:round_pushpin: 2. Скопируйте или перепишите в конфигурационный файл межсетевого экрана srv настройку трансляции адресов.

`@srv`: `/etc/nftables/nftables.nft`
```console
table inet filter {
       chain input {
               type filter hook input priority filter; policy accept;
       }

       chain forward {
               type filter hook forward priority filter; policy accept;
       }

       chain output {
               type filter hook output priority filter; policy accept;
       }
}
table ip masq {
       chain do_masq {
               type nat hook postrouting priority srcnat; policy accept;
               oifname "eth0" masquerade
       }
}
```

:information_source: Данное изменение читается так:
 + Создать новую таблицу с именем `masq` для обработки пакетов протокола `IPv4`;
 + В таблице создать цепочку `do_masq` типа "Преобразование" в привязкой к контрольной точке `Postrouting`, приоритет цепочки - `srcnat`;
 + На выходе с интерфейса `eth0` (`oif` - `Output InterFace`) проводить трансляцию исходящего адреса.

:round_pushpin: 3. С помощью _команд мониторинга сети_ проверьте возможность выхода в интернет с client

`@client`

### Ограничение доступа

:round_pushpin: 1. Опишите следующие правила для настройки ограничений доступа к srv

`@srv`
```console
[root@srv ~]# nft add table ip limiter
[root@srv ~]# nft add chain ip limiter netcatstop "{type filter hook input priority filter;}"
[root@srv ~]# nft add rule ip limiter netcatstop "tcp dport 1234 ct state new limit rate over 15/minute drop"
[root@srv ~]#
```

:information_source: Данные правила описывают следующие действия с межсетевым экраном:
+ В таблицу для пакетов протокола `IPv4` названия `limiter` добавить цепочку `netcatstop`;
 + Добавить в цепочку правило "Если по `TCP` на порт 1234 происходит более 15 новых подключений (`ct state new` - `Connection Tracker state == new`)  в минуту, сбрасывать пакеты"
